# FinanceBuddy: Data Model Design

## Overview

This document defines the complete database schema for FinanceBuddy v1. The data model supports zero-based envelope budgeting backed by true double-entry accounting, with the accounting layer invisible to users.

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Accounting model | True double-entry with category-as-virtual-account | Every transaction produces balanced journal entries (lines sum to zero). Categories map to virtual `expense`/`revenue` accounts in the ledger. Guarantees data integrity without exposing accounting concepts to users. |
| Monetary storage | `decimal(12,2)` | Exact decimal arithmetic native to Postgres. No cents-to-dollars conversion. Max $9,999,999,999.99. |
| Authentication | Rails 8 built-in (`bin/rails generate authentication`) | Lightweight, no extra gems. Sufficient for single-user/household app. |
| Opening balances | Equity account approach | System-managed "Opening Balances" equity account per ledger. Standard accounting practice, keeps the ledger balanced from the first entry. |
| Overspending policy | Deduct from Ready to Assign immediately | Modern YNAB convention. Forces honest budgeting. No schema impact — purely a computation rule over existing data. |
| Multi-tenancy | Ledger-based | A `ledgers` table sits between users and financial data. All financial tables reference `ledger_id`, not `user_id`. Enables future household access via `ledger_memberships` without migrating every table. |
| Budget allocations | Separate from accounting entries | Budget assignments express intent, not financial transactions. The budget layer reads from the ledger but does not write to it. Keeps the ledger clean and budgeting flexible. |
| On-budget flag | Boolean `on_budget` on accounts | Allows any account type to be on-budget or tracking-only (e.g., a savings cash account that's tracking-only). |
| Schema approach | Flat relational, no views | Simple tables with indexes. Views or materialized views can be added later if specific queries become pain points. Personal finance volumes (thousands of transactions) don't require optimization. |

### 2. Accounts

#### `accounts`

| Column | Type | Notes |
|--------|------|-------|
| `id` | `bigint` PK | |
| `ledger_id` | `bigint` FK | NOT NULL, indexed |
| `name` | `string` | NOT NULL |
| `account_type` | `string` | NOT NULL. One of: `cash`, `credit`, `loan`, `investment`, `equity`, `expense`, `revenue` |
| `on_budget` | `boolean` | NOT NULL, default `true`. When `false`, account is tracking-only |
| `balance` | `decimal(12,2)` | NOT NULL, default `0`. Cached running balance, updated transactionally |
| `cleared_balance` | `decimal(12,2)` | NOT NULL, default `0`. Cleared + reconciled transactions only |
| `reconciled_at` | `timestamp` | Last successful reconciliation |
| `display_order` | `integer` | NOT NULL, default `0` |
| `archived` | `boolean` | NOT NULL, default `false` |
| `created_at` | `timestamp` | |
| `updated_at` | `timestamp` | |

**Indexes:**
- `(ledger_id, account_type)` — listing accounts grouped by type
- `(ledger_id, archived)` — filtering active accounts

**Account types:**
- `cash`, `credit`, `loan`, `investment` — user-created real accounts
- `equity` — system-managed, one per ledger for opening balances
- `expense` — system-managed, one per category (virtual ledger account)
- `revenue` — system-managed, for income (virtual ledger account)

**Defaults for `on_budget`:** `cash` and `credit` → `true`; `loan`, `investment`, `equity`, `expense`, `revenue` → `false`. User can override for non-system account types.

System-managed account types (`equity`, `expense`, `revenue`) are filtered out of all user-facing account lists.

#### `transaction_lines`

The journal entry lines. Every transaction has 2+ lines that sum to zero.

| Column | Type | Notes |
|--------|------|-------|
| `id` | `bigint` PK | |
| `transaction_id` | `bigint` FK | NOT NULL |
| `account_id` | `bigint` FK | NOT NULL. Can be a real account or a category's virtual account |
| `amount` | `decimal(12,2)` | NOT NULL. Positive = debit, negative = credit |
| `memo` | `text` | Nullable. Per-line memo for splits |
| `created_at` | `timestamp` | |
| `updated_at` | `timestamp` | |

**Indexes:**
- `(transaction_id)` — load all lines for a transaction
- `(account_id)` — account register and balance queries

**Constraint:** `SUM(amount)` across all lines in a transaction must equal `0.00` (application-enforced).

#### How transaction modes map to lines

**Expense** ($52.30 from Chequing, categorized as Groceries):

| Line | Account | Amount |
|------|---------|--------|
| 1 | Chequing (cash) | -52.30 |
| 2 | Groceries (expense) | +52.30 |
| | **Sum** | **0.00** |

**Split expense** ($80 from Visa — $52.30 Groceries, $27.70 Household):

| Line | Account | Amount |
|------|---------|--------|
| 1 | Visa (credit) | -80.00 |
| 2 | Groceries (expense) | +52.30 |
| 3 | Household (expense) | +27.70 |
| | **Sum** | **0.00** |

**Income** ($3,200 paycheque to Chequing):

| Line | Account | Amount |
|------|---------|--------|
| 1 | Chequing (cash) | +3,200.00 |
| 2 | Income (revenue) | -3,200.00 |
| | **Sum** | **0.00** |

**Transfer** ($500 from Chequing to Savings):

| Line | Account | Amount |
|------|---------|--------|
| 1 | Chequing (cash) | -500.00 |
| 2 | Savings (cash) | +500.00 |
| | **Sum** | **0.00** |

**Opening balance** ($5,000 in Chequing):

| Line | Account | Amount |
|------|---------|--------|
| 1 | Chequing (cash) | +5,000.00 |
| 2 | Opening Balances (equity) | -5,000.00 |
| | **Sum** | **0.00** |

### 5. Budget Allocations

#### `budget_allocations`

| Column | Type | Notes |
|--------|------|-------|
| `id` | `bigint` PK | |
| `ledger_id` | `bigint` FK | NOT NULL |
| `category_id` | `bigint` FK | NOT NULL |
| `month` | `date` | NOT NULL. Always the 1st of the month |
| `assigned` | `decimal(12,2)` | NOT NULL, default `0` |
| `created_at` | `timestamp` | |
| `updated_at` | `timestamp` | |

**Indexes:**
- `(ledger_id, month)` — load entire budget grid for a month
- `(category_id, month)` unique — one allocation per category per month

**Computed values (not stored):**
- **Spent** = `SUM(amount)` from `transaction_lines` for the category's virtual account where transaction date falls within the month
- **Available** = all-time `SUM(assigned)` minus all-time spent for this category (cumulative envelope balance with automatic rollover)
- **Ready to Assign** = total income (all-time) minus total assigned across all categories (all-time). Category overspends reduce Ready to Assign immediately.

Rows are created on demand when a user first assigns money to a category for a given month.

#### `recurring_transactions`

| Column | Type | Notes |
|--------|------|-------|

| `entry_type` | `string` | NOT NULL. One of: `expense`, `income`, `transfer` |
| `amount` | `decimal(12,2)` | NOT NULL. Always positive |
| `memo` | `text` | Nullable |
| `frequency` | `string` | NOT NULL. One of: `weekly`, `biweekly`, `monthly`, `quarterly`, `yearly` |
| `start_date` | `date` | NOT NULL |
| `end_date` | `date` | Nullable. `null` = indefinite |
| `next_due_date` | `date` | NOT NULL |
| `auto_enter` | `boolean` | NOT NULL, default `false` |
| `created_at` | `timestamp` | |
| `updated_at` | `timestamp` | |

**Indexes:**
- `(ledger_id, next_due_date)` — upcoming/overdue queries
- `(ledger_id, auto_enter, next_due_date)` — batch job: auto-enter transactions due today

**Behaviour:**
- Daily Solid Queue job creates transactions for auto-enter rules where `next_due_date <= today`, then advances `next_due_date`.
- Non-auto-enter recurrences appear as upcoming entries in the register.
- No split support on recurring transactions for v1.

- `(ledger_id, fingerprint)` — fallback dedup
- `(matched_transaction_id)` — staging entries linked to existing transactions

## Foreign Key Cascades

| Parent | Child | On Delete |
|--------|-------|-----------|
| `ledger` | all financial tables | `CASCADE` |
| `transaction` | `transaction_lines` | `CASCADE` |
| `category_group` | `categories` | `RESTRICT` |
| `category` | `budget_allocations` | `RESTRICT` |
| `account` | `transaction_lines` | `RESTRICT` |
| `payee` | `transactions` | `NULLIFY` |
| `payee` | `payee_rules` | `CASCADE` |
